# DataFrame и Jupyter-интерфейс SQL-анализатора

## Цель

Дать пользователю готовый Jupyter Notebook, в который можно передать
`pandas.DataFrame` с колонками `query_text` и `query_text_template`, получить
построчный разбор значений/масок/regex и агрегаты по физическим таблицам,
колонкам и значениям. HTML является необязательным представлением, а не
основным результатом.

## Выбранный подход

Ноутбук остаётся тонким пользовательским интерфейсом. Проверяемая логика живёт
в `gp_sql_analyzer.dataframe`, а AST, lineage, классификация паттернов и HTML
переиспользуются из существующего пакета. Это предпочтительнее:

1. полностью самодостаточного ноутбука — он дублировал бы большой парсер и
   быстро разошёлся с тестируемой версией;
2. только прямого SQL-режима Greenplum — он не покрывает переданный пользователем
   DataFrame и затрудняет интерактивное исследование;
3. копирования промежуточных JSONL вручную — новый API делает этот шаг внутри.

Прямое чтение `analytics.query_log` остаётся доступно существующей CLI-командой и
показывается в ноутбуке как необязательный источник DataFrame.

## Входной контракт

Обязательный `queries_df`:

- `query_text` — исходный SQL;
- `query_text_template` — SQL с `&CHARACTER` внутри заменённых строковых
  литералов.

Необязательные колонки:

- `query_id` — идентификатор строки или запроса; при отсутствии создаётся из
  индекса DataFrame;
- `source_row_count` — вес предварительно сгруппированной пары; по умолчанию 1.

Необязательный `schema_df`:

- `table_schema`, `table_name`, `column_name`;
- `table_catalog` — если требуется четырёхчастное физическое имя.

Без `schema_df` AST и квалифицированные алиасом колонки продолжают
обрабатываться, но неквалифицированные колонки могут честно получить
`ambiguous` или `unresolved`. Пустая схема не должна приводить к догадкам.

## Python API

Основная функция:

```python
result = analyze_dataframe(
    queries_df,
    schema_df=schema_df,              # необязательно
    default_schema="public",
    dialect="postgres",
    placeholder="&CHARACTER",
    include_original_literals=True,
    include_null_checks=True,
    output_dir="reports/query_logs",  # необязательно
    build_html=False,
)
```

Результат содержит:

- `row_analysis_df` — ровно одна строка на каждую входную строку
  `queries_df`, в исходном порядке;
- `aggregate_df` — одна строка на комбинацию физической колонки, значения и
  места использования;
- `details_df` — дополнительное развёрнутое представление, одна строка на
  значение или NULL-проверку;
- `errors_df` — ошибки parse/align/lineage без остановки всего DataFrame;
- `catalog_columns_df` — одна строка на физическую колонку;
- `catalog_tables_df` — одна строка на физическую таблицу;
- `catalog_report` — иерархический словарь для дата-каталога и HTML;
- пути созданных артефактов, если был передан `output_dir`.

### `row_analysis_df`

Все исходные колонки `queries_df` сохраняются. Добавляются:

- `analysis` — список словарей с найденными значениями/условиями;
- `analysis_count` — число элементов списка;
- `resolved_count`, `multi_source_count`, `ambiguous_count`,
  `unresolved_count`;
- `analysis_status`: `ok`, `partial` или `error`;
- `analysis_errors` — список ошибок конкретного SQL.

Каждый элемент `analysis` содержит физические кандидаты, статус lineage,
`WHERE/JOIN_ON/HAVING/CASE/SELECT`, оператор, роль значения, исходный литерал,
извлечённое значение, шаблон, семейство/формат паттерна и источник события.
Количество и порядок строк `row_analysis_df` всегда совпадают с входным
DataFrame, даже если SQL ошибочен или не содержит значений.

### `aggregate_df`

Гранулярность одной строки:

```text
catalog_name + schema_name + table_name + column_name
+ extracted_value + clause_context + operator_or_function
+ value_role + pattern_family + pattern_format
```

В строке сохраняются `source_row_count`, `occurrence_count`,
`distinct_query_count`, доля значения внутри колонки и примеры `query_id`.
Только однозначно разрешённые физические колонки попадают в `aggregate_df`;
остальные события доступны в `row_analysis_df`, `details_df` и блоке quality.

API принимает объект, совместимый с pandas DataFrame, но импорт pandas является
необязательной зависимостью пакета. При отсутствии pandas выдаётся понятная
ошибка установки extras `gp-sql-analyzer[notebook]`.

## Источники условий

Детальная таблица объединяет три источника и помечает их полем `origin`:

1. `template` — точные пары исходного строкового литерала и `&CHARACTER`;
2. `original_literal` — числовые, строковые и выраженные границы из исходного
   SQL, которых нет среди уже сопоставленных placeholder-вхождений;
3. `null_check` — `IS NULL` и `IS NOT NULL`, для которых нет литерала и
   placeholder.

Совпавшие template/original события дедуплицируются по физическому lineage,
контексту, оператору и исходному литералу. Таким образом `%purple%` не
считается дважды, а `BETWEEN 1200 AND 1200 + 11` и `ss_addr_sk IS NULL`
сохраняются.

Для NULL-проверки используются:

- `operator_or_function`: `IS NULL` или `IS NOT NULL`;
- `value_role`: `null_check`;
- `raw_literal` и `extracted_value`: `NULL`;
- `pattern_family` и `pattern_format`: `null_check`.

Внешнее сравнение со скалярным подзапросом не должно превращать весь подзапрос
в значение. Его непосредственная константа (`0.9`) хранится отдельно, а
условия внутри подзапроса (`ss_store_sk = 4`, `ss_addr_sk IS NULL`) являются
отдельными событиями своих физических колонок.

## Подсчёт и качество

- `source_row_count` умножает частоту события и сохраняет статистику исходных
  строк DataFrame.
- В популярность физической колонки входят только `resolved` события с ровно
  одной колонкой.
- `multi_source`, `ambiguous` и `unresolved` сохраняются в `quality` и не
  раздаются кандидатам.
- Константы `SELECT 'catalog'` остаются `unresolved`, потому что у них нет
  физической колонки.
- `schema_df` добавляет в каталог все известные таблицы и колонки, включая
  колонки с нулевой статистикой.

## Ноутбук

`notebooks/sql_catalog_from_dataframe.ipynb` содержит исполняемые клетки:

1. импорт и конфигурация;
2. пример ожидаемой структуры `queries_df`;
3. необязательное построение `schema_df`;
4. запуск `analyze_dataframe`;
5. просмотр ошибок, таблиц и колонок;
6. примеры фильтрации топов, масок и regex;
7. ссылки на сохранённые JSON/JSONL и, только при `build_html=True`, HTML;
8. необязательный пример read-only чтения `analytics.query_log` и `pg_catalog`
   через существующие переменные окружения без паролей в ноутбуке.

Ноутбук поставляется выполненным на небольшом сложном fixture с CTE,
подзапросом, `IN`, LIKE, regex, арифметической границей и NULL-проверкой. Эти
данные явно подписываются как демонстрационные и легко заменяются реальным
`queries_df`.

## Ошибки и безопасность

- Отсутствующие обязательные колонки DataFrame вызывают `ValueError` с их
  именами.
- Пустой DataFrame создаёт пустые, но валидные результаты и полный инвентарь
  схемы.
- `build_html=True` без `output_dir` вызывает понятный `ValueError`, потому что
  HTML должен иметь целевой путь.
- Ошибка одного SQL записывается в `errors_df`; остальные строки продолжают
  обрабатываться.
- Notebook не выполняет анализируемый SQL и не пишет в Greenplum.
- Пример чтения из Greenplum использует read-only соединение и не содержит
  credentials.

## Критерии приёмки

1. DataFrame с обязательными колонками обрабатывается одной функцией.
2. `schema_df` необязателен; с ним lineage использует полные физические имена и
   добавляет нулевые колонки.
3. LIKE/regex, числовые значения, `BETWEEN`-выражения и NULL-проверки не
   дублируются и получают правильный контекст.
4. Случай Q44 представляется отдельными событиями для `ss_store_sk = 4` и
   `ss_addr_sk IS NULL`, без помещения всего подзапроса в поле значения.
5. Главные результаты возвращаются как `row_analysis_df` и `aggregate_df`;
   details/errors JSONL, иерархический JSON и колонковый JSONL записываются
   только при наличии `output_dir`.
6. HTML создаётся только при `build_html=True`; значение по умолчанию `False`.
7. Ноутбук выполняется от начала до конца без ручного редактирования на fixture.
8. Полный набор автоматических тестов проходит.
