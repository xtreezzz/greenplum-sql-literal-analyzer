# Greenplum-first notebook input design

## Цель

`notebooks/sql_catalog_from_dataframe.ipynb` должен быть самостоятельной точкой запуска для анализа таблицы Greenplum. Пользователь задаёт имя исходной таблицы и параметры окружения, запускает notebook сверху вниз и получает все рабочие DataFrame без ручной подготовки `queries_df`.

## Основной сценарий

1. Конфигурационная ячейка задаёт `SOURCE_TABLE`, `DEFAULT_SCHEMA`, `OUTPUT_DIR`, `BUILD_HTML` и параметры пакетной загрузки.
2. Подключение создаётся из `GP_HOST`, `GP_PORT`, `GP_DBNAME`, `GP_USER`, `GP_PASSWORD`, `GP_SSLMODE` и переводится в read-only режим.
3. Greenplum группирует строки по `query_text` и `query_text_template`. Число исходных строк сохраняется в `source_row_count`.
4. Notebook загружает полный снимок доступных таблиц и колонок из `pg_catalog` в `schema_df`.
5. `analyze_dataframe` строит SQLGlot AST, сопоставляет литералы с шаблонами и разрешает физические колонки.
6. Результаты раскрываются в отдельные переменные DataFrame и отображаются в notebook.

## Конфигурация

Обязательная настройка:

- `SOURCE_TABLE`: квалифицированное имя `schema.table`.

Основные необязательные настройки:

- `DEFAULT_SCHEMA`: схема для неквалифицированных таблиц в разбираемых SQL;
- `OUTPUT_DIR`: каталог JSON/JSONL-артефактов или `None`;
- `BUILD_HTML`: `False` по умолчанию;
- `BATCH_SIZE`: размер серверного курсора;
- `LIMIT`: ограничение для пробного запуска или `None`.

Параметры подключения не записываются в notebook и читаются только из окружения.

## Выходные DataFrame

После основной ячейки доступны:

- `queries_df`: уникальные пары исходного SQL и шаблона с `source_row_count`;
- `schema_df`: снимок каталога Greenplum;
- `row_analysis_df`: одна строка на уникальную пару SQL/шаблон;
- `details_df`: одно найденное употребление значения, маски или regex на строку;
- `aggregate_df`: агрегация по schema/table/column/value/context/operator/pattern;
- `catalog_tables_df`: статистика по таблицам;
- `catalog_columns_df`: статистика по колонкам;
- `errors_df`: изолированные ошибки разбора и lineage.

`source_row_count` сохраняет частоту физических строк исходной таблицы, поэтому агрегаты отражают исходный объём, хотя одинаковые пары разбираются один раз.

## Безопасность и ошибки

- Имя таблицы проходит существующую строгую проверку квалифицированных идентификаторов.
- Фильтры и лимиты не собираются из непроверенных значений.
- Сессия Greenplum read-only; notebook ничего не создаёт и не изменяет в базе.
- Ошибка подключения или отсутствующие переменные окружения завершают загрузочную ячейку с понятным сообщением.
- Ошибка отдельного SQL сохраняется в `errors_df` и не останавливает весь анализ.
- HTML создаётся только при `BUILD_HTML = True` и заданном `OUTPUT_DIR`.

## Структура notebook

Порядок ячеек:

1. описание и импорты;
2. конфигурация Greenplum;
3. подключение и загрузка `queries_df`;
4. загрузка `schema_df`;
5. одна явно обозначенная основная ячейка SQLGlot-анализа;
6. построчный, детальный и агрегированный DataFrame;
7. статистика дата-каталога и ошибки;
8. опциональный HTML.

Демонстрационный TPC-DS DataFrame не является основным путём и не должен подменять реальную загрузку. Для автоматических тестов Greenplum-вызовы проверяются тестовыми соединениями и структурными проверками notebook без обращения к реальной базе.

## Проверка

- Тест notebook подтверждает порядок: Greenplum load находится до `analyze_dataframe`.
- Тест проверяет наличие всех восьми выходных DataFrame-переменных.
- Существующие тесты Greenplum проверяют read-only соединение, безопасное имя таблицы, preaggregation и server-side batching.
- Полный набор unit-тестов должен проходить без реального Greenplum.
