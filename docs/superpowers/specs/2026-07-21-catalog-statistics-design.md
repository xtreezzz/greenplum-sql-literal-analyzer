# Статистика SQL-использования для дата-каталога

## Цель

Построить воспроизводимую статистику по каждой физической таблице и колонке,
которую затем можно загрузить в дата-каталог и просмотреть в автономном HTML.
Демонстрационный результат строится по всем 99 закреплённым запросам TPC-DS и
24 DDL-файлам. Тот же агрегатор должен принимать детальные строки, полученные
из `analytics.query_log(query_text, query_text_template)`, без повторного разбора SQL.

## Выбранная архитектура

Пайплайн разделён на три слоя:

1. Парсер создаёт нормализованные события использования литералов с lineage,
   контекстом, оператором, значением, маской и форматом регулярного выражения.
2. Постобработчик агрегирует события и инвентарь DDL в канонический объект
   каталожной статистики.
3. JSON, плоский JSONL и HTML генерируются только из этого объекта.

Так HTML не зависит от SQLGlot, а повторная агрегация для дата-каталога не
требует снова читать и разбирать исходные SQL-запросы.

## Состав статистики

В каталог включаются все таблицы и колонки из переданного снимка схемы, даже
если для них не нашлось однозначно связанного литерала. Для таких объектов
счётчики значений и условий равны нулю; это не утверждает, что колонка вообще
не упоминалась в SQL без литерала.

Для каждой таблицы сохраняются:

- полное имя, число колонок и число активных колонок;
- число условий, литералов и затронутых запросов;
- распределение по секциям SQL, операторам и семействам паттернов;
- вложенный список колонок.

Для каждой колонки сохраняются:

- полное физическое имя;
- число условий и литералов;
- число уникальных запросов;
- распределение по `WHERE`, `JOIN_ON`, `HAVING`, `CASE`, `SELECT` и другим
  обнаруженным контекстам;
- операторы и функции (`=`, `IN`, `BETWEEN`, `LIKE`, `ILIKE`, regex и другие);
- семейства и форматы масок/регулярных выражений;
- самые популярные значения с частотой, числом запросов и примерами query id;
- примеры запросов, где использовалась колонка.

Одна группа условия может содержать несколько значений, например
`sm_carrier IN ('DHL', 'BARIAN')`. В таком случае условие считается один раз,
а каждое значение получает собственную запись популярности. Выражение
`BETWEEN 1200 AND 1200 + 11` сохраняет две логические границы — `1200` и
`1200 + 11` — без ошибочного превращения арифметического выражения в `11`.

## Контракт файлов

### Иерархический JSON

`catalog-stats.json` содержит:

- `metadata`: версия формата, источник, commit, диалект, время генерации;
- `summary`: таблицы, колонки, активные колонки, запросы, условия, литералы и
  показатели качества lineage;
- `tables`: полный отсортированный список таблиц и колонок;
- `quality`: неоднозначные, multi-source и unresolved группы с причинами и
  примерами, которые не смешиваются с достоверной популярностью значений.

### Плоский JSONL

`catalog-columns.jsonl` содержит одну строку на физическую колонку и повторяет
её агрегаты в форме, удобной для пакетной загрузки в дата-каталог или
Greenplum. Вложенные топы и распределения остаются JSON-объектами/массивами.

### HTML

`catalog-stats.html` — автономный файл без CDN и внешних ресурсов. Он содержит:

- сводку покрытия и качества;
- поиск по схеме, таблице, колонке, значению, маске и оператору;
- фильтры по активности, секции SQL и типу паттерна;
- карточку каждой таблицы и строку каждой колонки;
- топ значений, масок и regex-форматов;
- отдельную секцию неоднозначных случаев;
- адаптивную и печатную вёрстку.

Визуальный язык продолжает существующий технический редакционный отчёт:
бумажный фон, тёмные чернила, кирпичный акцент и моноширинные SQL-значения.

## Правила доверия и подсчёта

- В статистику физической колонки попадает только lineage со статусом
  `resolved` и ровно одной колонкой.
- `ambiguous`, `multi_source` и `unresolved` учитываются в `quality`, но не
  увеличивают популярность конкретной колонки.
- Для строк `details.jsonl` частоты взвешиваются по `source_row_count`, чтобы
  предварительная агрегация одинаковых запросов не теряла исходную частоту.
- Для TPC-DS каждый SQL-файл имеет вес 1.
- Сортировка детерминирована: сначала частота по убыванию, затем значение или
  имя по возрастанию.
- Топы ограничиваются на уровне представления; полные распределения секций,
  операторов и семейств сохраняются.

## Ограничения

- TPC-DS демонстрирует корректность на сложных аналитических запросах, но не
  является статистикой реальной таблицы `analytics.query_log`.
- Без отдельного снимка схемы постобработчик может агрегировать уже разрешённые
  колонки из `details.jsonl`, но не может добавить отсутствующие колонки с
  нулевыми счётчиками.
- Отчёт описывает статическое использование SQL, а не частоту выполнения из
  журналов Greenplum, если `source_row_count` не передаёт эту частоту.

## Критерии приёмки

1. Все 99 TPC-DS запросов разбираются, все 24 DDL-таблицы и все их колонки
   присутствуют в JSON, JSONL и HTML.
2. Для каждой колонки с найденными литералами доступны значения, маски/regex,
   операторы и секции SQL; остальные колонки явно имеют нулевые счётчики и не
   ошибочно помечаются как полностью неиспользуемые.
3. Неоднозначный lineage отделён от доверенной статистики.
4. JSONL содержит ровно одну строку на колонку и может быть построен из
   агрегированного объекта без разбора SQL.
5. HTML строится только из каталожного объекта, работает без сети и позволяет
   найти конкретную таблицу, колонку или значение.
6. Автоматические тесты, полный прогон корпуса и визуальная проверка HTML
   проходят без ошибок.
